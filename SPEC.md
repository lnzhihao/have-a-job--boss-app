# BOSS 直聘 Auto-Agent — 实现规格书（交接 Claude Code）

> 这份文档是给 Claude Code 的完整需求规格。请阅读全文后，基于已有的 Electron 骨架（`src/main.js` + `src/index.html`）和参考脚本（`reference/boss_agent.py`），实现一个**可实时看到运行状态**的桌面 App。

---

## 一、项目目标

做一个常驻 macOS 桌面的悬浮窗口 App，自动化 BOSS 直聘求职投递。核心是**两个独立流程**，且必须能在界面上**实时看到每一步在做什么**（不是模拟，是真实浏览器自动化的实时回显）。

技术栈：**Electron（已有骨架）+ Playwright（真实浏览器自动化）+ Claude API（评分/回复）**。

---

## 二、两个流程（核心架构）

### 流程一：打招呼（慢节奏，批量）

```
扫职位列表 → 三层筛选 → AI 评分 → 发打招呼 → 随机延迟 → 下一个
```

| 步骤 | 说明 |
|------|------|
| 1. 搜索 | 按关键词在 BOSS 搜索职位（深圳） |
| 2. 区域筛选 | 职位地址必须含：南山 / 前海 / 宝安，否则跳过 |
| 3. 薪资筛选 | 薪资上限 < 15k 直接跳过；< 20k 降优先级 |
| 4. 关键词筛选 | 职位名含目标关键词（见第四节）才进入评分 |
| 5. AI 评分 | 调 Claude，0-100 分，< 60 跳过 |
| 6. 发打招呼 | 点"立即沟通"，发对应话术模板 |
| 7. 延迟 | 随机 8-15 秒，避免操作过快被风控 |
| 8. 上限 | 每日最多 25 次（BOSS 免费沟通约 30/天） |

### 流程二：HR 回复后秒发简历（快节奏，抢注意力窗口）⭐ 重点

```
实时监听 IM → 检测到 HR 新回复 → 立刻发简历附件 + 作品集链接 → 标记已发 → 退出，交给用户
```

**这个流程的核心是"快"**。HR 刚回复你的那几分钟注意力最集中，要在这个窗口内把简历直接砸过去，让 HR 看到消息就能立刻点开简历看完——这是转化率最高的时机。之后 Agent 不再介入聊天，由用户本人接手沟通。

| 要求 | 细节 |
|------|------|
| 监听方式 | 轮询 IM 页面（建议 5-10 秒一次）或监听 DOM 变化，检测哪些会话有 HR 新消息 |
| 触发条件 | 某个**我已打过招呼**的会话，HR 首次回复了（区分"我发的"和"HR 发的"） |
| 响应速度 | 检测到后**立即**执行发送，目标延迟 < 10 秒 |
| 发送内容 | ① 简历附件（PDF，通过 IM 的附件上传功能）② 作品集链接（一条文字消息）|
| 防重复 | 每个会话只发一次，记录到本地 `sent_resume.json` |
| 发完即止 | 发送后**停止自动回复**，标记该会话为"已交接"，剩下让用户手动聊 |
| 桌面通知 | 发送成功后弹 macOS 通知："已向 [公司] 发送简历，请接手沟通" |

> ⚠️ 技术难点：BOSS IM 的附件上传是流程二最难的部分。如果通过 Playwright 操作 `<input type=file>` 上传不稳定，退而求其次：**先发一条带作品集链接 + 简历在线预览链接的文字消息**（秒级、稳定），附件上传作为增强项。优先保证"快"和"稳"。

## 三、实时界面要求（"可实时看到的页面"）⭐

这是用户这次最在意的点。要求：

1. **真实回显，非模拟**：界面上的日志、计数、状态，全部来自 Playwright 真实执行的事件，通过 IPC 从主进程实时推送到渲染进程。

2. **实时日志流**：每一个动作（搜索/筛选/评分/投递/监听到HR回复/发简历）即时打印到日志区，带时间戳和颜色（成功绿/警告黄/跳过灰）。

3. **实时指标**：已投递、HR回复数、已发简历数、跳过数，随事件实时跳动。

4. **可选：嵌入实时浏览器画面**。两种方案二选一：
   - 方案 A（简单）：Playwright 用 `headless: false` 启动有头浏览器，用户能直接看到浏览器在自动操作。
   - 方案 B（进阶）：定时截图（`page.screenshot()`）推送到 Electron 窗口里显示一个缩略实时画面。
   - 建议先做方案 A，省事且直观。

5. **两个流程独立开关**：界面上「流程一：打招呼」和「流程二：监听发简历」是两个独立的启停按钮，可以单独跑，也可以同时跑。

实时通信用 Electron 的 `ipcRenderer` / `ipcMain` 即可，主进程跑 Playwright，每个事件 `win.webContents.send('agent-event', {...})` 推给界面。

## 七、技术架构建议

```
┌─────────────────── Electron 主进程 (main.js) ───────────────────┐
│                                                                  │
│  Playwright (chromium, headless:false)                           │
│    ├── 流程一 worker：搜索→筛选→评分→打招呼                       │
│    └── 流程二 worker：轮询 IM →检测HR回复→发简历                  │
│                                                                  │
│  每个动作 → win.webContents.send('agent-event', {...})           │
│                                                                  │
│  Claude API 调用（评分 / 智能回复）                              │
│  本地存储：config.json / sent_resume.json / cookies.json         │
└──────────────────────────┬───────────────────────────────────────┘
                           │ IPC
┌──────────────────────────▼───────────────────────────────────────┐
│              渲染进程 (index.html) — 悬浮窗口 UI                  │
│   实时日志 · 实时指标 · 两流程开关 · 规则配置 · 设置             │
└──────────────────────────────────────────────────────────────────┘
```

关键实现点：
1. **登录态**：首次 `headless:false` 让用户手动扫码，保存 cookies，后续复用。
2. **流程一 / 流程二 用同一个浏览器上下文**（共享登录态），但跑在不同的异步任务里，可独立启停。
3. **会话状态机**：每个会话标记 `greeted`（已打招呼）/ `replied`（HR已回）/ `resume_sent`（已发简历）/ `handed_over`（已交接），存本地，重启不丢。
4. **选择器要容错**：BOSS 的 DOM class 会变，每个关键元素准备 2-3 个备选 selector，参考 `boss_agent.py` 里的写法。
5. **反风控**：随机延迟、模拟人类输入速度（`delay: 30`）、不要并发狂点。

## 十、注意事项与风险

- **账号风险**：自动化操作可能触发 BOSS 风控，建议控制频率、加随机延迟，必要时让用户确认每次投递。
- **附件上传**：流程二的简历附件上传是最不稳定的环节，做好降级方案（纯文字 + 链接）。
- **选择器维护**：BOSS 前端会变，selector 需要容错和易于更新。
- **这是给本人求职用的工具**，所有操作模拟真人行为，不做规模化滥用。
